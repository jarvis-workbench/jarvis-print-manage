# 命名规范与规划制度（长期约束）

更新时间：2026-03-28

## 1. 命名规范（项目级）
- 代码文件命名采用模块化命名，不采用纯语义化命名。
- 同目录避免后缀冗余命名：
  - 例如 `electron/worker/` 下使用 `printer-state.mjs`，不再使用 `printer-state-worker.mjs`。
- 打印机相关脚本采用统一模块前缀：
  - `printer-list-installed.scps1`
  - `printer-install-from-backup.scps1`
  - `printer-uninstall.scps1`

## 2. 记忆落盘制度
- 任何“用户明确要求记住”的项目约束，必须在 `docs/` 落盘。
- 落盘范围不受现有文件限制，可按主题新增 `.md` 文件。
- 每次功能阶段完成并提交后，至少同步更新：
  - `docs/07-开发与升级日志.md`
  - 受影响的规范文档（如 `docs/03`、`docs/06`、本文件）

## 3. 规划制度
- 当用户提到“规划”：
  - 先做可实施方案（产品/技术节点、范围、风险、验收）；
  - 先与用户讨论并确认；
  - 未经确认不进入编码实施。
- 规划文档统一放在 `docs/planning/`，每个规划使用独立文件夹，编号从 `00001-...` 连续增长。
- 单个规划的扩展文档必须在该规划文件夹内继续编号（`00001/00002/...`），不得和其他规划穿插。
- 规划文档必须显式声明“关联规划”，并在关联规划变更时同步回填优化项。
