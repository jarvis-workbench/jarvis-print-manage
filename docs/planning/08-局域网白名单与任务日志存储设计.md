# 存储设计草案：局域网白名单与任务日志（P2/P3）

更新时间：2026-03-28  
状态：草案（待 P1 完成后评审）  
关联文档：
- `docs/planning/04-局域网共享模块拆分草案.md`
- `docs/planning/06-局域网共享IPC与数据契约草案.md`

## 1. 目标
- 固化白名单与任务日志的本地存储结构，避免 P2 开发时重复改 schema。
- 保证日志可按 `taskId` 追溯完整链路，且支持失败复盘。

## 2. 存储位置建议
- 根目录：`<userData>/lan/`
- 文件：
  - `node-whitelist.json`：配对节点白名单
  - `lan-task-log.jsonl`：任务事件日志（JSON Lines）
  - `lan-state.json`：运行态快照（可选）

## 3. 白名单结构（`node-whitelist.json`）
```json
{
  "version": 1,
  "updatedAt": "2026-03-28T03:40:00.000Z",
  "nodes": [
    {
      "nodeId": "node-uuid",
      "machineName": "PC-A",
      "firstPairedAt": "2026-03-28T03:00:00.000Z",
      "lastAcceptedAt": "2026-03-28T03:35:00.000Z",
      "pairMethod": "one-time-code",
      "status": "trusted"
    }
  ]
}
```

约束：
- `nodeId` 唯一。
- `status` 首版仅允许：`trusted | blocked`。
- 删除节点时写审计事件到 `lan-task-log.jsonl`。

## 4. 任务日志结构（`lan-task-log.jsonl`）
每行一个事件：
```json
{
  "ts": "2026-03-28T03:41:00.000Z",
  "taskId": "lan-task-uuid",
  "event": "TRANSFERRING",
  "nodeId": "node-uuid",
  "offerId": "node-uuid:printer-key",
  "progress": 42,
  "errorCode": "",
  "errorMessage": "",
  "meta": {
    "archiveSize": 1234567,
    "retry": 0
  }
}
```

事件建议集合：
- `DISCOVERING_STARTED`
- `NODE_VALIDATED`
- `OFFER_FETCHED`
- `TRANSFER_STARTED`
- `TRANSFER_PROGRESS`
- `TRANSFER_FINISHED`
- `INSTALL_STARTED`
- `INSTALL_FINISHED`
- `TASK_FAILED`
- `TASK_CANCELED`

## 5. 写入与清理策略
- 写入策略：追加写（append-only），失败不覆盖历史。
- 轮转策略：
  - 单文件超过 20MB 时滚动到 `lan-task-log-YYYYMMDD.jsonl`。
  - 保留最近 14 天，超期按天删除。
- 异常保护：写入失败不影响主流程，但需上报 `LAN_LOG_WRITE_FAILED`。

## 6. IPC 对齐
- `lan:get-task`：优先从运行态缓存返回，再补读日志构建详情。
- `lan:cancel-task`：写入 `TASK_CANCELED` 事件，触发文件清理。
- `lan:list-nodes`：输出前过滤 `status = blocked` 节点。

## 7. 安全与隐私边界
- 日志禁止记录配对码明文。
- 日志中的 IP 可保留（用于排障），但不记录完整系统用户名。
- 提供“清空局域网记录”入口：清空白名单 + 历史日志。

## 8. 评审通过条件
- 白名单读写具备 schema 兜底与坏文件自愈（重建空结构）。
- 任务日志可完整重放一个失败案例并定位失败阶段。
- 清理策略验证通过，不误删当天活跃日志文件。
