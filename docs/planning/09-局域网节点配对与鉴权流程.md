# 流程图与时序：局域网节点配对与鉴权（P2）

更新时间：2026-03-28  
状态：草案（待 P1 完成后评审）  
关联文档：

## 1. 目标
- 固化首版“一次性配对码 + 白名单”流程，明确每一步输入、输出、失败处理。
- 限制首版复杂度：仅同网段 IPv4，不做跨网段中继与 TLS。

## 2. 角色定义
- 发起端（Requester）：请求查看远端能力或安装打印机的节点。
- 提供端（Provider）：提供打印机能力与驱动包的节点。

## 3. 配对状态机
- `UNPAIRED`：未配对。
- `PAIRING_PENDING`：收到配对请求待确认。
- `TRUSTED`：已写入白名单，可访问受限接口。
- `BLOCKED`：被拉黑，拒绝请求。

状态转换：
- `UNPAIRED -> PAIRING_PENDING`：收到有效配对请求。
- `PAIRING_PENDING -> TRUSTED`：配对码校验通过且用户确认。
- `PAIRING_PENDING -> UNPAIRED`：超时或用户拒绝。
- `TRUSTED -> BLOCKED`：用户手动拉黑。
- `BLOCKED -> TRUSTED`：用户解除拉黑并重新确认。

## 4. 时序（首版）
1. Requester 调用 `lan:pair:request`，携带 `nodeId/machineName/nonce`。
2. Provider 弹出确认 UI，并显示一次性配对码（6 位数字，120 秒有效）。
3. Requester 输入配对码，调用 `lan:pair:confirm`。
4. Provider 校验：
- `nonce` 未过期
- 配对码匹配
- `nodeId` 未被 `blocked`
5. Provider 写入 `node-whitelist.json`（`status=trusted`），回传 `pairToken`（短期令牌）。
6. 后续 `lan:list-offers/lan:request-install` 必须附带 `pairToken` 或白名单命中。

## 5. IPC 草案（新增）
- `lan:pair:request`
  - 入参：`{ nodeId: string machineName: string }`
  - 返回：`{ pairingId: string expiresAt: string }`
- `lan:pair:confirm`
  - 入参：`{ pairingId: string code: string }`
  - 返回：`{ status: 'trusted' token: string expiresAt: string }`
- `lan:pair:revoke`
  - 入参：`{ nodeId: string }`
  - 返回：`{ status: 'blocked' }`

## 6. 错误码建议
- `LAN_PAIR_EXPIRED`：配对窗口超时。
- `LAN_PAIR_CODE_INVALID`：配对码错误。
- `LAN_PAIR_BLOCKED`：节点已被拉黑。
- `LAN_PAIR_RATE_LIMITED`：短时间多次尝试。

## 7. 安全约束
- 配对码只显示后 120 秒内有效，失败 5 次自动失效。
- `pairToken` 默认 24 小时失效，失效后需白名单重新握手刷新。
- 日志中仅记录配对结果，不记录配对码明文。

## 8. 验收点
- 未配对节点无法获取 `offers`。
- 配对成功后可稳定调用 `lan:list-offers`。
- 拉黑后旧 token 立即失效。
