# 规范：局域网测试数据生成与命名（P2/P3）

## 1. 目标
- 统一测试数据来源与命名，避免联调阶段“样本不一致”导致误判。
- 为异常注入、回归、发布 Gate 提供可复用数据集。

## 2. 数据目录建议
- 根目录：`<repo>/tests/fixtures/lan/`
- 子目录：
  - `nodes/`：节点样本
  - `offers/`：远端能力样本
  - `tasks/`：任务轨迹样本
  - `logs/`：jsonl 日志样本
  - `archives/`：模拟压缩包元数据（不存真实大文件）

## 3. 命名规范
- 文件命名：`<domain>-<scenario>-<seq>.json`
- 示例：
  - `node-trusted-001.json`
  - `offer-arch-mismatch-002.json`
  - `task-retry-success-003.json`
- 日志样本：`lan-log-<scenario>.jsonl`

## 4. 样本最小字段

### 4.1 节点样本（`nodes/*.json`）
```json
{
  "nodeId": "node-a-001",
  "machineName": "PC-A",
  "arch": "x64",
  "servicePort": 24821,
  "status": "trusted"
}
```

### 4.2 Offer 样本（`offers/*.json`）
```json
{
  "offerId": "node-a-001:printer-001",
  "driverName": "HP Universal Printing PCL 6",
  "archiveFormat": "pdrv.zip",
  "archiveSha256": "<hex>",
  "archiveSize": 1234567
}
```

### 4.3 Task 样本（`tasks/*.json`）
```json
{
  "taskId": "lan-task-001",
  "status": "TRANSFERRING",
  "retryCount": 1,
  "maxRetry": 3,
  "errorCode": ""
}
```

## 5. 场景覆盖要求
必须至少包含：
- `happy-path`：正常成功链路。
- `pair-failed`：配对失败。
- `hash-mismatch`：哈希不一致。
- `retry-success`：重试后成功。
- `retry-exhausted`：重试耗尽失败。
- `rollback-recover`：回滚后恢复。

## 6. 数据生成流程
1. 从基础模板复制（`base-node/base-offer/base-task`）。
2. 仅改当前场景相关字段，不改无关字段。
3. 生成后执行 schema 校验（后续脚本实现）。
4. 记录样本清单到 `tests/fixtures/lan/manifest.json`。

## 7. 版本与维护
- 每次 schema 变更必须同步更新 `manifest.json` 的 `version`。
- 过期样本不删除，移动到 `deprecated/` 并记录替代样本。
