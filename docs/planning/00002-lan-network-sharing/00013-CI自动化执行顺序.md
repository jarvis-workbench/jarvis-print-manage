## 1. 目标
- 固化 CI 阶段顺序，避免“先跑慢测后挂在类型检查”的低效流程。
- 将 P2/P3 放行标准映射到可执行流水线步骤。

## 2. 建议流水线阶段
1. `lint-and-typecheck`
2. `unit-tests`
3. `contract-tests`（IPC/数据结构）
4. `fault-injection-tests`（FI-01~FI-06）
5. `rollback-drill-tests`
6. `artifact-and-report`

## 3. 每阶段输入输出

### 3.1 `lint-and-typecheck`
- 输入：代码 + `src/env.d.ts`
- 输出：静态检查报告
- 阻断条件：任一错误即停止

### 3.2 `unit-tests`
- 输入：核心模块单测
- 输出：覆盖率报告
- 阻断条件：失败率 > 0

### 3.3 `contract-tests`
- 输入：`tests/fixtures/lan/` + IPC mock
- 输出：契约一致性报告
- 阻断条件：字段缺失/错误码不匹配

### 3.4 `fault-injection-tests`
- 输入：`fault-injection.json` + 场景矩阵
- 输出：`fault-injection-report.json`
- 阻断条件：FI-01~FI-06 任一失败

### 3.5 `rollback-drill-tests`
- 输入：回滚演练脚本/模板
- 输出：`rollback-drill-report.json`
- 阻断条件：回滚后本地链路异常

### 3.6 `artifact-and-report`
- 输入：前述报告
- 输出：汇总报告 + 构建产物
- 阻断条件：无（只归档）

## 4. 执行顺序原则
- 快速失败优先：越早越便宜的检查越靠前。
- 场景测试并行：`fault-injection-tests` 内部可并行，但结果统一汇总。
- 回滚演练串行：避免并行污染环境。

## 5. 门槛建议
- 类型检查：0 error。
- 单测通过率：100%。
- 契约测试通过率：100%。
- 异常注入通过率：100%。
- 回滚演练：核心步骤全通过。

## 6. 失败处置
- CI 失败自动附带失败阶段、场景 ID、关键日志路径。
- 对于 `fault-injection-tests`，必须输出首个失败场景的上下文事件序列。
- 对于 `rollback-drill-tests`，必须输出回滚前后状态 diff。

## 7. 发布前映射
- 仅当 `1~5` 阶段全部通过，才允许进入发布审批。
- 发布审批需引用：
  - `fault-injection-report.json`
  - `rollback-drill-report.json`
  - `gate-summary.md`
