# 协作约定（用户规则）

本文件记录“用户明确提出的协作规则”，后续维护必须遵守。

## Git 提交规则（高优先级）
- 只有用户明确说“提交 git”时才提交。
- 提交信息（commit message）使用中文。
- 不要擅自改写历史（如 `--amend`、强制重置等），除非用户明确要求。
- 每次提交 git 前，应优先确认/排除本机环境配置类文件（例如 `electron/resource/config/system.json`）是否需要提交。

## 沟通与执行规则
- 用户偏好“直接落地修改 + 快速验证”，不要只给方案不动手。
- 涉及 UI/交互改动时，优先遵循用户提出的具体风格要求。
- 多轮迭代后，必须把关键决策同步写入 `docs/`。
- “规划”类需求不直接写代码；先输出可落地方案并落盘文档，待用户确认后再实施开发。

## 记忆规则
- `docs/` 是长期记忆来源，未来对话应优先读取。
- 新需求若改变既有规则，必须更新本文件并注明日期。
- 每次一段功能开发结束并提交后，必须同步更新 `docs/`（至少更新日志与受影响约定）。
- 用户明确提出“记住”的项目约束，必须落盘到 `docs/`。
- 记忆落盘不限制于既有文档，可在 `docs/` 新增 `.md` 文件。

## 命名规则
- 代码文件命名采用模块化命名，不采用纯语义化命名。
- 同目录命名避免后缀冗余；例如 `electron/worker/` 内文件不再附加 `-worker` 后缀。
- 脚本命名按模块前缀聚类（例如 `printer-*`），便于排序与维护。

## 规划文档规则
- 规划文档固定目录：`docs/planning/`。
- 每个规划单独一个 `.md` 文件，使用语义化中文文件名。
- 规划文档需覆盖：目标、范围、分阶段节点、接口/数据结构、风险与回滚、验收标准。
